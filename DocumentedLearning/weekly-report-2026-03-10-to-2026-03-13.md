# CAPS Mobile Weekly Progress Report

March 10-13, 2026

## Executive Summary

From March 10 to March 13, 2026, CAPS Mobile work focused on improving mobile usability, strengthening visual consistency, and reducing deployment risk. The team completed system-wide dark mode support, added subject image uploads, refined the student dashboard and leaderboard experience, improved Android responsiveness and orientation handling, centralized theme management. Two visible frontend features are now prepared but still depend on backend completion: leaderboard data integration and social login.

## Progress by Date

### March 10, 2026

- Completed system-wide dark mode support across authentication flows, dashboards, and shared UI components.
- Added subject image upload capability for Admin and Faculty users, including backend validation, image compression, and replacement of previous subject images.
- Updated the student subject card flow so uploaded subject images appear in the dashboard with a fallback when no custom image is available.
- Outcome: the app became more adaptable for low-light use and more flexible for subject presentation without changing the core user flow.

### March 11-12, 2026

- Redesigned the student dashboard with a swipeable insights carousel, full-width subject presentation, and a more balanced centered layout for mobile screens.
- Performed follow-up dashboard adjustments to keep the redesign compatible with both light and dark themes, improve card border readability, and make tap navigation cues clearer.
- Added responsive spacing refinements, responsive text scaling, and a floating inset student bottom navigation to reduce clipping and improve mobile alignment.
- Implemented student leaderboard UI enhancements, including a podium layout, filter controls for program and subject, and dynamic tab labeling.
- Refined social login presentation on the login screen with higher-fidelity Google and Facebook branding.
- Added portrait-lock protections and micro-interactions to improve mobile consistency and the tactile feel of card interactions.
- Outcome: the student-facing experience became more polished and mobile-friendly
### March 13, 2026

- Centralized app theme colors into CSS custom properties so major color changes can be managed from one place instead of being scattered across the frontend.
- Removed the user-facing server configuration modal and moved environment selection control into source configuration for a more predictable controlled deployment setup.
- Fixed Android-facing responsiveness issues affecting the login page, practice exam submit button, header spacing, and page layout padding.
- Prepared the leaderboard page to consume backend data through the planned API contract, including support for optional program and subject filters.
- Added frontend handlers for Google and Facebook social login routes so the login experience is ready once backend OAuth endpoints are available.
- Outcome: maintenance overhead was reduced, deployment behavior became more controlled, and key frontend integrations moved closer to completion.

## Key Learnings and Decisions

- Centralized theme management reduces maintenance cost because visual changes can be made in one place instead of across hundreds of hardcoded color references.
- Hardcoded server configuration is the better fit for this controlled deployment model because it improves predictability and removes the risk of users selecting the wrong backend.
- Frontend-first delivery for backend-dependent features allows visible progress to continue while backend contracts are completed in parallel.
- Mobile-safe spacing, theme-safe redesign choices, and responsive typography are necessary to avoid regressions across different Android device sizes and both theme modes.


## Risks / Pending Dependencies

- Leaderboard integration is still incomplete until the backend `GET /api/leaderboard` endpoint is implemented and returns the expected filtered data.
- Social login is still incomplete until the backend Google and Facebook OAuth redirect endpoints are implemented.
- Backend recovery and deployment processes must continue to preserve the correct `APP_KEY` whenever an existing database volume is reused.
